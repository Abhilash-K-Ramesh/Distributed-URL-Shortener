# Distributed URL Shortener with Rate Limiting

A URL shortener built to demonstrate backend systems concepts — caching, distributed ID generation, rate limiting, and horizontal scaling — rather than just CRUD basics.

## Status

🚧 **In progress** — Week 1 (core API + caching) complete. Rate limiting, Dockerization, and horizontal scaling are up next.

## Tech Stack

- **API:** FastAPI (Python)
- **Database:** PostgreSQL
- **Cache:** Redis
- **Dev Environment:** GitHub Codespaces
- **Planned:** Docker, Docker Compose, Nginx (load balancing), k6/Locust (load testing)

## Architecture (current)

```
Client → FastAPI app → Redis (cache, checked first)
                     ↘ PostgreSQL (source of truth, on cache miss)
```

On a cache miss, the app queries Postgres, returns the result, and populates Redis with a 1-hour TTL so subsequent requests for the same short code are served without touching the database.

## Features Implemented So Far

### Core URL shortening
- `POST /shorten` — accepts a long URL, checks for an existing short code to avoid duplicates, and generates a new one if needed.
- `GET /{short_code}` — looks up the short code and issues an HTTP redirect to the original URL, or a 404 if not found.
- **Base62 encoding** — short codes are generated by encoding the row's auto-increment ID into a compact alphanumeric string (`0-9`, `a-z`, `A-Z`), rather than using a random hash.

### Redis read-through caching
- `GET /{short_code}` checks Redis first. On a miss, it falls back to Postgres and writes the result into Redis (`TTL = 3600s`) for future requests.
- **Benchmark:** cold lookup (Postgres) averaged **~19ms**; cached lookups (Redis) averaged **~2ms** — roughly a **9x speedup** for cached URLs.

### Configuration & secrets
- Database and Redis connection details are loaded from a `.env` file via `python-dotenv`, kept out of version control via `.gitignore`.

## Setup & Running Locally (GitHub Codespaces)

1. **Start Postgres and Redis containers**
   ```bash
   docker start pg-urlshortener redis-urlshortener
   # If they don't exist yet:
   docker run --name pg-urlshortener -e POSTGRES_PASSWORD=devpass -e POSTGRES_DB=urlshortener -p 5432:5432 -d postgres:16
   docker run --name redis-urlshortener -p 6379:6379 -d redis:7
   ```

2. **Create the database table** (first time only)
   ```bash
   docker exec -it pg-urlshortener psql -U postgres -d urlshortener -c "
   CREATE TABLE urls (
       id SERIAL PRIMARY KEY,
       long_url TEXT NOT NULL,
       short_code VARCHAR(10) UNIQUE,
       created_at TIMESTAMP DEFAULT NOW()
   );"
   ```

3. **Set up the Python environment**
   ```bash
   cd app
   python3 -m venv venv
   source venv/bin/activate
   pip install fastapi uvicorn psycopg2-binary redis python-dotenv
   ```

4. **Create a `.env` file** in `app/`
   ```
   DB_PASSWORD=devpass
   DB_HOST=localhost
   DB_PORT=5432
   DB_NAME=urlshortener
   DB_USER=postgres
   ```

5. **Run the server**
   ```bash
   uvicorn main:app --reload --port 8000
   ```

6. **Test it**
   ```bash
   curl -X POST http://localhost:8000/shorten \
     -H "Content-Type: application/json" \
     -d '{"long_url": "https://example.com/some/long/path"}'

   curl -L http://localhost:8000/<short_code>
   ```

   Interactive API docs are available at `/docs` (Swagger UI) once the server is running.

## Design Decisions

- **Base62 over random hashing:** encoding the auto-increment ID keeps short codes compact and collision-free without needing a separate uniqueness check on insert.
- **Read-through caching over write-through:** the cache only populates for short codes that are actually visited, avoiding wasted memory on links nobody clicks.
- **`.env` for secrets from day one:** avoids retrofitting configuration management later as the project grows.

## Known Limitations / Backlog

- Each request currently opens a new Postgres connection; a connection pool (`psycopg2.pool`) would reduce overhead in production.
- No rate limiting yet (in progress — see Roadmap).
- Single app instance, no containerization yet.
