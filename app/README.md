# Distributed URL Shortener with Rate Limiting

A URL shortener built to demonstrate backend systems concepts — caching, distributed rate limiting, atomic operations, horizontal scaling, and load balancing — rather than just CRUD basics.

## Status

✅ Core API, caching, rate limiting, Dockerization, horizontal scaling, and load testing all complete.

## Tech Stack

- **API:** FastAPI (Python)
- **Database:** PostgreSQL
- **Cache / Rate Limiter state:** Redis
- **Reverse Proxy / Load Balancer:** Nginx
- **Containerization:** Docker, Docker Compose
- **Load Testing:** k6
- **Dev Environment:** GitHub Codespaces

## Architecture

```
                        ┌────────────┐
 Client ───────────────▶│   Nginx    │  (single entry point, port 8000)
                        └─────┬──────┘
                              │ round-robin
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
         ┌────────┐     ┌────────┐      ┌────────┐
         │ app-1  │     │ app-2  │      │ app-3  │   (FastAPI, stateless)
         └───┬────┘     └───┬────┘      └───┬────┘
             │              │               │
             └──────┬───────┴───────┬───────┘
                     ▼               ▼
               ┌──────────┐    ┌──────────┐
               │  Redis   │    │ Postgres │
               │ (cache + │    │ (source  │
               │  limits) │    │ of truth)│
               └──────────┘    └──────────┘
```

All three app instances are stateless and share the same Redis and Postgres — this is what makes caching and rate limiting behave correctly under load balancing, rather than each instance keeping its own (incorrect) local state.

## Features Implemented

### Core URL shortening
- `POST /shorten` — accepts a long URL, checks for an existing short code to avoid duplicates, and generates a new one if needed.
- `GET /{short_code}` — looks up the short code and issues an HTTP redirect to the original URL, or a 404 if not found.
- **Base62 encoding** — short codes are generated by encoding the row's auto-increment ID into a compact alphanumeric string (`0-9`, `a-z`, `A-Z`).

### Redis read-through caching
- `GET /{short_code}` checks Redis first. On a miss, it falls back to Postgres and writes the result into Redis (`TTL = 3600s`) for future requests.
- **Benchmark:** cold lookup (Postgres) averaged **~19ms**; cached lookups (Redis) averaged **~2ms** — roughly a **9x speedup** for cached URLs.

### Distributed rate limiting
- Per-(method, path, client IP) limits enforced via Redis, e.g. `POST /shorten` limited to 5 requests / 10s, `GET /{short_code}` limited to 20 requests / 10s.
- **Race condition found and fixed:** an initial GET-then-INCR implementation was provably unsafe under concurrency — 10 simultaneous requests against a limit of 5 all succeeded when the check-then-act gap was artificially widened. Fixed by moving the check-and-increment into a single **atomic Redis Lua script**, confirmed to correctly cap concurrent requests at the configured limit afterward.
- **Verified correct across multiple instances:** because limiter state lives in shared Redis (not per-instance memory), rate limits remain correctly enforced even when requests are load-balanced across 3 separate app containers.

### Horizontal scaling & load balancing
- App containerized with Docker; Postgres, Redis, and the app orchestrated together via Docker Compose.
- Nginx added as a reverse proxy / load balancer in front of multiple app instances (`docker-compose up --scale app=3`), round-robining requests across them.
- **Distributed-safe ID generation verified:** Postgres's `SERIAL` sequence is atomic at the database engine level, so concurrent inserts from 3 separate app instances never produced duplicate short codes (confirmed via `GROUP BY ... HAVING COUNT(*) > 1` returning zero rows under concurrent load).

### Load testing (k6)
- Simulated realistic read-heavy traffic (redirect requests), ramping from 10 to 50 concurrent virtual users over 40 seconds.
- **With rate limiting active:** 100% of requests returned an expected status (`307` or `429`); ~91% were correctly rate-limited under sustained multi-user load, confirming the limiter holds up under real concurrent traffic, not just manual testing.
- **With rate limiting temporarily raised (raw throughput test):** 0% failures across 914 requests. Average latency **5.63ms**, p90 **11.16ms**, p95 **15.72ms**, sustained ~22.6 req/s (paced by the test script's think-time, not a hard ceiling).

### Configuration & secrets
- Database and Redis connection details loaded from environment variables (`.env` locally, Compose environment block in containers), kept out of version control via `.gitignore`.

## Setup & Running Locally

1. **Bring up the full stack** (Postgres, Redis, Nginx, and 3 app instances)
   ```bash
   docker-compose up --build --scale app=3
   ```

2. **Create the database table** (first time only)
   ```bash
   docker-compose exec postgres psql -U postgres -d urlshortener -c "
   CREATE TABLE urls (
       id SERIAL PRIMARY KEY,
       long_url TEXT NOT NULL,
       short_code VARCHAR(10) UNIQUE,
       created_at TIMESTAMP DEFAULT NOW()
   );"
   ```

3. **Test it**
   ```bash
   curl -X POST http://localhost:8000/shorten \
     -H "Content-Type: application/json" \
     -d '{"long_url": "https://example.com/some/long/path"}'

   curl -L http://localhost:8000/<short_code>
   ```

   Interactive API docs are available at `/docs` (Swagger UI) once the stack is running.

4. **Run the load test**
   ```bash
   k6 run loadtest.js
   ```

## Design Decisions

- **Base62 over random hashing:** encoding the auto-increment ID keeps short codes compact and collision-free without a separate uniqueness check on insert.
- **Read-through caching over write-through:** the cache only populates for short codes that are actually visited, avoiding wasted memory on links nobody clicks.
- **Redis Lua script over plain GET/INCR for rate limiting:** guarantees the check-and-increment happens as a single atomic operation, closing a real race condition that was reproduced and confirmed during development.
- **Per-endpoint rate limits:** write-heavy endpoints (`/shorten`) are limited more strictly than read-heavy endpoints (`/{short_code}`), mirroring how real APIs are typically tiered.
- **Postgres SERIAL for ID generation over a custom distributed counter:** the database sequence is already atomic across concurrent connections, so it safely handles ID generation from multiple app instances without extra coordination logic.

## Known Limitations / Backlog

- Each request currently opens a new Postgres connection; a connection pool (`psycopg2.pool`) would reduce overhead in production.
- Secrets (`POSTGRES_PASSWORD`) are set directly in `docker-compose.yml` for local development; a real deployment would source these from a `.env` file via `env_file:` or a secrets manager.
- Rate limit identity is based on client IP only — no API key/user-based limiting yet.
- No HTTPS/TLS termination configured in Nginx (fine for local dev, would be required for production).

## Engineering Notes

A few things worth calling out that came up during development, since they reflect real debugging rather than a frictionless build:
- Diagnosed and fixed a race condition in the rate limiter by artificially widening the check-then-act window with a temporary delay, confirming the bug, then fixing it with an atomic Lua script and re-confirming under the same test.
- Learned the difference between GitHub Codespaces' internal `127.0.0.1` and the externally forwarded port URL — a common source of "connection refused" errors early in development.
- Resolved several `psycopg2.OperationalError` issues caused by Postgres containers not persisting across Codespace restarts, which motivated moving from manually-managed containers to Docker Compose for consistency.
